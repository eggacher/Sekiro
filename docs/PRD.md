## 1. 产品概述

### 1.1 项目背景

中后台应用在登录鉴权、权限管理、CRUD 表格、布局 UI 等基础能力上存在大量重复开发，团队每次新立项都要从零搭建，造成效率损耗与质量参差。Sekiro 旨在沉淀一套**开箱即用、规范统一、可复用**的中后台脚手架，让业务团队专注于核心业务逻辑。

### 1.2 产品定位

一个**面向中后台场景、自带管理后台的半成品脚手架**。它本身就是一个可运行的项目（含示例管理后台），开发者在它的基础上做减法与扩展，即可快速产出自己的业务系统。

**一句话描述**：开箱即用的中后台脚手架 —— 登录就能用，改改就能上线。

### 1.3 核心价值

| 价值点 | 说明 |
| --- | --- |
| ⏱ 提效 | 开箱即用，省去 80% 基础建设时间 |
| 🧩 规范 | 统一目录约定、代码风格、API 结构，降低协作成本 |
| 🔒 安全 | 内置 RBAC、加密、防注入等安全基线 |
| 🚀 可扩展 | 模块化设计，按需启用/裁剪 |
| 📚 可维护 | 完善文档 + 种子数据，上手即用 |

### 1.4 目标用户与使用场景

| 用户角色 | 使用场景 |
| --- | --- |
| 全栈/后端开发者 | 基于脚手架搭建新业务系统的后台 |
| 前端开发者 | 复用布局、组件、请求封装快速开发页面 |
| 技术负责人 | 作为团队统一的技术规范与项目模板 |
| 初学者 | 学习中后台项目的标准工程实践 |

### 1.5 设计原则

1. **约定优于配置**：明确的目录与命名约定，减少决策成本。
2. **合理默认**：所有功能提供可用默认值，零配置即可启动。
3. **渐进增强**：MVP 可用，P1/P2 按需开启，不强制依赖。
4. **可裁剪**：模块低耦合，业务侧可自由删减。
5. **技术栈无关的抽象**：核心概念稳定，不与具体框架绑定。

### 4.1 认证与授权模块

#### F-AUTH-01 登录与登出 ｜ P0

- **需求描述**：用户通过账号密码登录系统，获取访问凭证；支持安全登出。
- **用户故事**：作为管理员，我希望登录后能进入后台，登出后凭证立即失效。
- **详细需求**：
  - 登录页：账号、密码输入框，含显隐密码、记住我；
  - 登录成功后下发 Token，跳转至默认首页；
  - 凭证存储于 localStorage / cookie（可配置）；
  - 登出后清除本地凭证并跳转登录页，服务端使该 Token 失效。
- **验收标准**：
  - ✅ 正确账号密码可登录成功；
  - ✅ 错误密码有明确错误提示；
  - ✅ 登出后无法使用旧 Token 访问受保护接口。

#### F-AUTH-02 会话与 Token 管理 ｜ P0

- **需求描述**：统一管理用户会话，支持 Token 过期自动刷新。
- **详细需求**：
  - 支持 JWT 或 Session 方案（二选一，可配置）；
  - Access Token 短有效期，配合 Refresh Token 无感续期；
  - Token 过期 / 失效时，前端统一拦截并跳转登录页。
- **验收标准**：续期对用户无感知；Token 失效后不会出现接口报错刷屏。

#### F-AUTH-03 路由守卫 ｜ P0

- **需求描述**：前端路由级鉴权，未登录或无权限时拦截跳转。
- **详细需求**：
  - 全局前置守卫校验登录态；
  - 无权限访问受保护路由时跳转 403 页；
  - 未登录跳转登录页并携带 redirect。
- **验收标准**：直接访问任意内部 URL，未登录均跳转登录页。

#### F-AUTH-04 RBAC 权限模型 ｜ P0

- **需求描述**：基于"用户-角色-权限"三元模型实现访问控制。
- **详细需求**：
  - 数据模型：`user` ∎ `role` ∎ `permission`，多对多关系；
  - 后端中间件做接口级权限校验（`@RequirePermission`）；
  - 前端页面级（路由 meta）与按钮级（`v-permission` 指令）控制。
- **验收标准**：无权限用户看不到对应菜单/按钮，且直接调用接口被后端拒绝（403）。

#### F-AUTH-05 动态路由与菜单 ｜ P1

- **需求描述**：根据用户权限动态生成前端路由与菜单。
- **详细需求**：
  - 登录后拉取当前用户菜单树；
  - 前端根据菜单树注册路由、渲染侧边栏；
  - 后台菜单管理可配置路由 path、组件、图标、排序。
- **验收标准**：不同角色登录看到不同菜单；后台改菜单后刷新生效。

#### F-AUTH-06 验证码登录 ｜ P1

- 登录页支持图形验证码；预留短信/邮箱验证码扩展位。
- **验收标准**：错误验证码不可登录；防暴力重试。

#### F-AUTH-07 第三方登录（OAuth） ｜ P1

- 支持 GitHub / Google / 微信 等第三方账号登录与账号绑定。

#### F-AUTH-08 修改密码 / 密码强度 ｜ P1

- 个人中心修改密码，校验新密码强度（长度、字符组合）。

#### F-AUTH-09 多因素认证 MFA ｜ P2

- 支持 TOTP（如 Google Authenticator）二次验证。

#### F-AUTH-10 SSO 单点登录 ｜ P2

- 支持 SAML / OIDC 协议对接企业身份系统。

### 4.2 用户与组织模块

#### F-USER-01 用户管理 ｜ P0

- **需求描述**：管理员维护系统用户。
- **详细需求**：
  - 列表：分页、多条件搜索（用户名/手机号/状态）、排序；
  - 操作：新增、编辑、删除、批量删除、启用/停用、重置密码、分配角色；
  - 字段：用户名、昵称、邮箱、手机、部门、角色、状态、创建时间。
- **验收标准**：CRUD 完整闭环；停用用户无法登录。

#### F-USER-02 角色管理 ｜ P0

- **需求描述**：维护角色及其权限。
- **详细需求**：
  - 角色 CRUD；为角色分配菜单/权限（树形勾选）；
  - 一个角色可对应多个用户；可查看角色下用户列表。
- **验收标准**：调整角色权限后，相关用户权限即时生效。

#### F-USER-03 菜单 / 权限管理 ｜ P0

- **需求描述**：维护菜单树与权限项（按钮权限）。
- **详细需求**：
  - 树形菜单管理（目录/菜单/按钮三级）；
  - 配置路由、组件路径、图标、是否缓存、是否可见；
  - 支持启用/停用。
- **验收标准**：菜单变更后具备该菜单权限的用户可见。

#### F-USER-04 个人中心 ｜ P0

- 查看个人资料；修改昵称、头像、基本信息；修改密码。

#### F-USER-05 部门 / 组织架构 ｜ P1

- 树形部门管理；用户归属部门；支持数据权限按部门范围控制。

#### F-USER-06 数据权限 ｜ P1

- 角色可配置数据范围：全部、本部门、本部门及以下、自定义、仅本人。
- **验收标准**：列表查询结果受数据权限约束。

#### F-USER-07 用户导入导出 ｜ P1

- 支持 Excel 模板下载、批量导入（含校验失败提示）、列表导出。

#### F-USER-08 岗位管理 ｜ P1

- 岗位 CRUD，用户可关联岗位。

### 4.4 通用业务组件模块

> 该模块是脚手架的核心复用价值所在，沉淀高频业务组件。

#### F-COMP-01 标准 CRUD 表格 ｜ P0

- **需求描述**：封装搜索栏 + 表格 + 分页 + 新增/编辑弹窗的一体化组件。
- **详细需求**：
  - 通过配置（columns + search + form schema）声明式渲染；
  - 内置分页、loading、空状态、操作列；
  - 与请求层打通，自动管理查询参数。
- **验收标准**：一个业务页面 50 行配置即可完成完整 CRUD。

#### F-COMP-02 表单封装 ｜ P0

- **需求描述**：统一的表单组件，支持校验、布局、字段联动。
- **详细需求**：
  - Schema 驱动渲染；内置常用校验；
  - 支持栅格布局、分组、联动显隐；
  - 与 UI 组件库打通。
- **验收标准**：复杂业务表单无需手写大量模板。

#### F-COMP-03 请求封装 ｜ P0

- 统一封装 HTTP 客户端：请求/响应拦截、Token 注入、错误处理、loading、重试。
- 响应统一解析 `{ code, message, data }` 结构。

#### F-COMP-04 消息提示 ｜ P0

- 封装 toast / 确认弹框 / 通知，统一风格与调用方式。

#### F-COMP-05 文件 / 图片上传 ｜ P1

- 支持拖拽、多文件、预览、限制；存储后端可切换本地 / OSS / S3。

#### F-COMP-06 富文本编辑器 ｜ P1

- 集成成熟富文本编辑器（如 TinyMCE / WangEditor / Tiptap）。

#### F-COMP-07 字典 / 枚举下拉 ｜ P1

- 结合数据字典自动渲染下拉、标签；支持缓存。

#### F-COMP-08 图标选择器 ｜ P1

- 内置常用图标库选择，菜单配置时使用。

#### F-COMP-09 导入导出工具 ｜ P1

- 通用 Excel 导入导出封装（前后端）。

#### F-COMP-10 低代码表单设计器 ｜ P2

- 可视化拖拽生成表单 schema。

#### F-COMP-11 图表封装 ｜ P2

- ECharts 常用图表的封装组件。

### 4.8 后端基础设施模块

#### F-BE-01 ORM 与数据迁移 ｜ P0

- ORM 集成；提供迁移脚本与版本管理。

#### F-BE-02 统一响应结构 ｜ P0

- 所有接口返回 `{ code, message, data }`；分页统一 `{ list, total, page, pageSize }`。

#### F-BE-03 全局异常处理 ｜ P0

#### F-BE-06 种子数据 ｜ P0

#### F-BE-08 文件存储抽象 ｜ P1

#### F-BE-10 接口限流 / 防刷 ｜ P1

## 5. 非功能需求

| 维度 | 指标 / 要求 |
| --- | --- |
| **性能** | 首屏 LCP < 2.5s（中等网络）；接口 P95 < 300ms（常规查询）；列表千行渲染流畅 |
| **并发** | 默认支撑 500 QPS（单机），可通过水平扩展 |
| **可用性** | 无单点强依赖（Redis 可降级）；核心接口错误率 < 0.1% |
| **兼容性** | 浏览器：Chrome/Edge/Firefox/Safari 近两年版本；最小屏幕 1280×720 |
| **安全** | 通过 OWASP Top 10 自检；密钥不入库；接口默认鉴权 |
| **可维护性** | 代码规范统一；核心模块单测覆盖 > 60%；关键逻辑有注释 |
| **可观测性** | 关键链路有日志；健康检查可用；预留监控接入点 |
| **国际化** | 文案全部走 i18n，支持至少中英双语 |
| **可扩展性** | 模块松耦合，P2 功能可插拔 |

## 8. 风险与依赖

| 风险 / 依赖 | 影响 | 应对 |
| --- | --- | --- |
| 技术栈未定 | 影响实现细节与依赖选型 | 优先确认前后端技术栈 |
| 权限模型设计不当 | 后期业务改造成本高 | MVP 即采用标准 RBAC，预留数据权限扩展 |
| 第三方登录/MFA 依赖外部服务 | P1/P2 阻塞 | 放到后续版本，先保证账号密码闭环 |
| 代码生成器复杂度高 | 工期风险 | P1 中后期再做，先沉淀组件 |
| 多环境 / 部署差异 | 上线受阻 | 用 Docker 统一运行时，提供完整示例 |
